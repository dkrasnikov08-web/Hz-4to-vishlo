# Защита секретов при деплое на Linux VPS

Краткий чек-лист. Подробности — в комментариях `deploy.sh` и `bybit_bot.service`.

## 1. Файлы, которые НЕЛЬЗЯ коммитить в git
- `.env` — реальные ключи. В репозитории должен быть только `.env.example` с плейсхолдерами.
- `*.log` — может содержать данные о балансе/позициях.
- Проверьте `.gitignore` — он уже добавлен и закрывает эти случаи.

**Если `.env` с реальными ключами уже когда-либо попадал в git** — смена ключей на Bybit обязательна, даже если коммит позже удалили (история git хранит старые версии файлов, `git rm` это не отменяет).

## 2. Перенос ключей на сервер
Не передавайте `.env` через команды, которые попадают в shell-историю или логи CI/CD (`scp` с паролем в команде, `echo "KEY=xxx" >> .env` в публичном Ansible-плейбуке и т.п.). Безопасные варианты:
- `scp` по SSH-ключу (без пароля в команде) + ручное редактирование на сервере (`nano .env`)
- Секрет-менеджер (Vault, AWS Secrets Manager, итд.), если используется

## 3. Права доступа на сервере
```bash
chown bybit_bot:bybit_bot /opt/bybit_bot/.env
chmod 600 /opt/bybit_bot/.env
```
`deploy.sh` делает это автоматически и **остановит запуск**, если права не 600 или в файле остались placeholder-значения (`your_key_here`).

## 4. Процесс запускается не от root
`bybit_bot.service` запускает бота от системного пользователя `bybit_bot` без shell и без sudo-прав (`useradd --shell /usr/sbin/nologin`). Даже если в коде или зависимости (`pybit`) найдётся RCE-уязвимость, ущерб ограничен правами этого пользователя.

## 5. Секреты не попадают в systemd-журнал
`EnvironmentFile=/opt/bybit_bot/.env` в unit-файле передаёт переменные процессу напрямую — они не отображаются в `systemctl show`, `systemctl cat` (только путь к файлу) и не дублируются в journald.

## 6. Защита от утечки через DEBUG-логи pybit
`logger_setup.py` теперь жёстко ограничивает уровень логов `pybit` до WARNING, независимо от `LOG_LEVEL` в `config.py`. Это защищает от случая, когда кто-то поставит `LOG_LEVEL=DEBUG` для отладки и не заметит, что pybit при DEBUG может писать подписанные HTTP-запросы (с подписью запроса, не самим секретом, но это всё равно чувствительные данные) в общий лог-файл.

## 7. systemd hardening (доп. слой)
В `bybit_bot.service` включены `ProtectSystem=strict`, `ProtectHome=true`, `NoNewPrivileges=true` и другие ограничения — даже скомпрометированный процесс не может писать за пределы `/opt/bybit_bot` или повышать привилегии.

## 8. Testnet → Mainnet
`BYBIT_TESTNET=true` должен оставаться включённым до тех пор, пока вы не убедитесь в стабильной работе бота. `deploy.sh` не переключает этот флаг автоматически — это осознанное ручное действие в `.env`.

## 9. API-ключ на бирже — права доступа
На стороне Bybit создавайте API-ключ с минимально необходимыми правами:
- ✅ Contracts — Read/Write (нужно для ордеров)
- ❌ Withdraw — НЕ включайте право на вывод средств. Боту это не нужно, а в случае утечки ключа это резко ограничит ущерб.
- Если Bybit позволяет — привяжите ключ к конкретному IP сервера (whitelist).

## Быстрый старт
```bash
# На сервере:
git clone <repo> /tmp/bybit_bot_src
cd /tmp/bybit_bot_src
sudo bash deploy.sh          # первый запуск — скрипт остановится и попросит создать .env
sudo nano /opt/bybit_bot/.env   # вписать реальные ключи
sudo bash deploy.sh          # повторный запуск — выставит права, поставит unit
sudo systemctl enable --now bybit_bot.service
sudo journalctl -u bybit_bot.service -f
```
